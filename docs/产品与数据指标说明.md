# AI 员工管理与价值展示平台 — 产品与数据指标说明

## 1. 产品概述

AI Workforce Platform 是一个内部管理看板，用于管理一支由 24 名 AI 员工组成的虚拟团队。团队分为三条业务线：管理团队、设计师团队、生产团队。平台通过可视化看板展示 AI 员工的工作状态、绩效指标和价值产出，帮助管理层实时掌握 AI 团队运营情况。

**技术栈：** Next.js 16, React 19, TypeScript, Tailwind CSS v4, shadcn/ui, SQLite + Drizzle ORM, ECharts, React Flow

---

## 2. 功能模块

### 2.1 驾驶舱 (/dashboard)

- **综合运营指数仪表盘**：弧形仪表盘，显示 0-100 的综合指数。计算公式：`round(((adoptionRate + accuracyRate) / 2) × 100)`。颜色：≥80 绿色，60-79 黄色，<60 红色。
- **团队状态概览**：3 个团队卡片，显示各团队健康率（在岗/总员工数）。点击可展开团队抽屉，查看成员列表、等级、经验值。
- **6 个 KPI 卡片**：
  1. 本月任务量（单位：个）
  2. 工具采用率（单位：%）
  3. 任务准确率（单位：%）
  4. 节省人工时（单位：h）
  5. 节省人力成本（前缀 ¥，单位：元）
  
  每个 KPI 卡片均含环比趋势箭头和迷你折线图（sparkline）。

- **团队效能对比**：分组柱状图，展示近 5 个月数据，每团队 3 条系列。点击可联动过滤热力图。
- **实时排行榜**：按经验值（XP）展示 Top 5 员工，可切换本周/本月。显示等级徽章、XP 值、名次变化。
- **AI 员工近 30 天活跃热力图**：以员工 × 日期为轴展示已完成任务数。点击格子可打开对应员工详情。
- **成就动态**：近期成就解锁动态信息流。
- **最近任务动态**：最新 8 条任务，带实时更新指示器。

### 2.2 AI 花名册 (/roster)

- 员工卡片网格，支持团队 Tab 筛选（全部/管理/设计/生产）和关键词搜索。
- 每张卡片显示：AI 生成肖像、姓名、职位、状态徽章、团队色左边框、3 个指标（本月任务/采纳率/准确率）。
- 点击卡片打开详情弹窗（Dialog），包含 4 个 Tab：
  - **档案**：基本信息 + IP 人设卡（年龄、MBTI、性格标签、口头禅、背景故事、工作方式、兴趣爱好、穿搭风格、视觉特征）。
  - **技能**：按类别分组展示技能，每项技能含星级评分（1-5）及按期统计指标。
  - **指标**：4 个 KPI 卡片 + ECharts 趋势图（任务数/节省工时/采用率/准确率）。
  - **版本**：版本日志时间线，含变更说明。

### 2.3 生产看板 (/production)

- **顶部 4 张统计卡片**：今日任务 / 完成率 / 执行中 / 平均质量分。
- **3 个 Tab 页签**：
  - **实时看板**：每 15 秒自动轮询运行中任务，支持团队筛选。每张任务卡片上显示迷你 SOP 步骤条。
  - **数据面板**：按日趋势图（按团队分组）、任务类型分布、质量仪表盘、员工排名。
  - **历史记录**：分页表格，支持搜索/团队/状态筛选，列：任务名称 / 执行者 / 类型 / 状态 / 质量分 / 开始时间 / 耗时。
- **任务详情弹窗**：4 项统计条（质量评分/执行耗时/重试次数/预估费用）+ 3 个 Tab（执行步骤/产出内容/执行反思）。

### 2.4 组织架构 (/org)

- React Flow 画布，层级布局：根节点 → 团队负责人 → 管理层 → 员工。
- 节点特性：团队色边框、SVG 进度环、状态点、等级/MBTI/连续活跃天数徽章。
- 控制项：团队筛选、缩放、自适应视图。
- 图例：团队颜色、状态点含义、节点尺寸说明。

### 2.5 系统设置 (/settings)

- **3 个 Tab 页签**：
  - **员工管理**：员工 CRUD，状态管理（上岗/下岗），头像生成。
  - **指标基准配置**：三级配置体系（全局基准/团队覆盖/员工覆盖），管理各任务类型的 `humanBaseline`（人工基准工时）和 `costPerHour`（时薪）。
  - **数据指标管理**：员工绩效/技能指标/任务数据的增删改查表格，支持下钻和导出（CSV/Excel）。

## 4. 数据指标详解

### 4.1 核心运营指标

| 指标名称 | 英文标识 | 单位 | 取值范围 | 计算公式 | 说明 |
|---|---|---|---|---|---|
| 综合运营指数 | operationalIndex | 分 | 0-100 | `round(((adoptionRate + accuracyRate) / 2) × 100)` | 衡量 AI 团队整体运营水平 |
| 本月任务量 | taskCount | 个 | ≥0 | 当月完成任务数汇总 | 按月统计 |
| 工具采用率 | adoptionRate | % | 0-100 | 原始值为 0.0-1.0，展示时 ×100 | 衡量 AI 工具在业务中的渗透率 |
| 任务准确率 | accuracyRate | % | 0-100 | 原始值为 0.0-1.0，展示时 ×100 | 衡量 AI 任务的正确性 |
| 节省人工时 | humanTimeSaved | 小时 | ≥0 | seed: `taskCount × 2.5 × random(0.8-1.2)` | 对比人工完成同等任务所需时间 |
| 节省人力成本 | savedCost | 元 (CNY) | ≥0 | `humanTimeSaved × costPerHour` | `costPerHour` 默认 46.875 元（¥375/天 ÷ 8小时） |
| 任务成功率 | successRate | % | 0-100 | `completedTasks / totalTasks × 100` | 全时段任务完成比例 |
| 完成率 | completionRate | % | 0-100 | `completedTasks / totalTasks × 100` | 按时间范围过滤的完成比例 |
| 团队健康度 | healthRate | % | 0-100 | `activeEmployees / totalEmployees × 100` | 在岗率 |

### 4.2 任务质量指标

| 指标 | 标识 | 范围 | 说明 |
|---|---|---|---|
| 质量评分 | qualityScore | 0-100（seed: 70-97） | 单任务质量打分；运行中任务为 null |
| Token 用量 | tokenUsage | 整数（seed: 1000-8000） | LLM token 消耗量 |
| 预估费用（API） | estimatedCost | USD | `tokenUsage × 0.00015` |
| 预估费用（UI 展示） | estimatedCost | 元 (CNY) | `tokenUsage × 0.0001` |
| 重试次数 | retryCount | 0-2（seed，80% 为 0） | 任务执行期间的重试次数 |
| 执行耗时 | duration | 分钟 | `actualEndTime - startTime`（seed: 10-70 分钟） |

### 4.3 技能指标

| 指标 | 标识 | 范围 | 说明 |
|---|---|---|---|
| 调用次数 | invocationCount | ≥0 | 技能在该月的调用总数 |
| 成功率 | successRate | 0.0-1.0 | 技能调用成功比例 |
| 平均响应时间 | avgResponseTime | 秒 | 单次调用的平均耗时 |
| 技能等级 | level | 1-5 | 熟练度星级评分 |

### 4.4 游戏化指标

| 指标 | 说明 |
|---|---|
| 经验值 (XP) | 任务 XP = `50 + 50 × (qualityScore / 100)`，范围 50-100/任务；连续活跃 XP：前 6 天 10 XP/天，第 7 天起 20 XP/天 |
| 等级 (Level) | 升级阈值：`200 × 1.5^(N-1)`。Level 1: 200 XP，Level 2: 300 XP，Level 3: 450 XP，以此类推 |
| 等级称号 | Level 1-3: 新手🌱，Level 4-6: 熟练⚡，Level 7-9: 精英🔥，Level 10-12: 大师💎，Level 13+: 传奇👑 |
| 连续活跃天数 (streak) | 连续有已完成任务的天数 |

### 4.5 成就系统

共 8 个成就，由系统自动检测并解锁：

| 成就名称 | 英文键 | 触发条件 |
|---|---|---|
| 首个满分 | first_perfect | `qualityScore === 100` |
| 七日之焰 | seven_flame | `streak >= 7` 天 |
| 闪电手 | flash_hand | 单日完成 ≥5 个任务 |
| 月度 MVP | mvp | 当月 XP 最高 |
| 完美主义 | perfectionist | 最近 10 个任务 `qualityScore` 均 >90 |
| 全能选手 | all_rounder | 技能数 ≥5 |
| 团队之星 | team_star | 完成 ≥3 种不同类型任务 |
| 成长飞速 | fast_growth | 当月等级提升 ≥3 |

## 5. 指标配置体系

### 5.1 三级配置优先级

系统采用三级配置结构，优先级从高到低：

```
员工级覆盖（employeeId ≠ null）
    ↓ 未命中
团队级覆盖（employeeId = null，team ≠ null）
    ↓ 未命中
全局基准（employeeId = null，team = null）
```

解析接口 `GET /api/metric-configs/resolve?taskType=&employeeId=` 会依次瀑布查找，返回最终生效的配置。

### 5.2 默认全局基准

| 任务类型 | 人工基准工时（小时） | 时薪（元/小时） | 备注 |
|---|---|---|---|
| 项目审计 | 4（管理团队覆盖：5h） | 46.875 | Audit |
| 绩效评估 | 3（管理团队覆盖：4h） | 46.875 | |
| 版本管理 | 2 | 46.875 | |
| 需求文档 | 8 | 46.875 | |
| 人员盘点 | 6 | 46.875 | |
| 剧本创作 | 3（生产团队覆盖：4h） | 46.875 | |
| 资源入库 | 0.5 | 46.875 | |
| 质量检查 | 0.25（生产团队覆盖：0.5h） | 46.875 | |
| 音频制作 | 2 | 46.875 | |
| 字幕制作 | 1.5 | 46.875 | |

### 5.3 成本计算公式

- **时薪基准**：46.875 元/小时（¥375/天 ÷ 8小时）
- **节省人力成本** = 节省人工时 × 时薪
- **Token 费用（API 层）** = `tokenUsage × $0.00015`（美元）
- **Token 费用（UI 展示）** = `tokenUsage × ¥0.0001`（人民币）

## 6. 任务工作流（SOP）

### 6.1 任务类型

按团队归类的任务类型：

**管理团队（7 种）：**
项目审计、绩效评估、版本管理、生产管理、业务分析、人员盘点、激励申报

**设计师团队（4 种）：**
战略规划、需求文档、软件设计、产品方案

**生产团队（9 种）：**
需求确认、生产评审、质量检查、资源入库、剧本创作、角色设计、图像生成、音频制作、字幕制作

### 6.2 SOP 步骤模板

每种任务类型均有固定的标准化步骤序列：

| 任务类型 | SOP 步骤序列 |
|---|---|
| 项目审计 | 数据采集 → 指标分析 → 风险评估 → 审计报告生成 → 审核校验 |
| 绩效评估 | 数据汇总 → 指标计算 → 评估报告 → 结果校验 |
| 版本管理 | 变更收集 → 影响分析 → 版本打包 → 发布验证 |
| 生产管理 | 任务分配 → 进度追踪 → 异常处理 → 结果汇总 |
| 业务分析 | 数据采集 → 清洗处理 → 分析建模 → 报告生成 |
| 人员盘点 | 信息采集 → 数据核对 → 缺口分析 → 盘点报告 |
| 激励申报 | 绩效汇总 → 资格审核 → 激励计算 → 申报提交 |
| 战略规划 | 趋势分析 → 方案制定 → 可行性评估 → 规划输出 |
| 需求文档 | 需求收集 → 分析整理 → 文档编写 → 评审修订 |
| 软件设计 | 架构分析 → 概要设计 → 详细设计 → 文档输出 |
| 产品方案 | 市场调研 → 方案设计 → 原型制作 → 方案评审 |
| 需求确认 | 需求解读 → 可行性分析 → 排期评估 → 确认输出 |
| 生产评审 | 质量检查 → 标准对照 → 问题标注 → 评审报告 |
| 质量检查 | 规范检查 → 内容审核 → 质量评分 |
| 资源入库 | 资源解析 → 格式校验 → 标签标注 → 入库归档 |
| 剧本创作 | 主题分析 → 大纲生成 → 场景描写 → 对白编写 → 文档输出 |
| 角色设计 | 角色定义 → 视觉概念 → 形象生成 → 细节调整 |
| 图像生成 | 需求解析 → 素材匹配 → 图像生成 → 质量检查 |
| 音频制作 | 文本解析 → 音色匹配 → 语音合成 → BGM 匹配 → 音频输出 |
| 字幕制作 | 文本解析 → 时间轴对齐 → 样式排版 → 输出导出 |

### 6.3 步骤状态流转

```
pending → running → completed
                 ↘ failed
                 ↘ skipped
```

**约束规则：** 运行中的任务有且仅有一个步骤处于 `running` 状态，该步骤之前的所有步骤均为 `completed`，之后的步骤均为 `pending`。

### 6.4 任务反思（Reflection）

`tasks.reflection` 字段存储 JSON 结构（非纯文本），格式如下：

```json
{
  "problems": ["发现的问题..."],
  "lessons": ["踩过的坑..."],
  "improvements": ["改进建议..."]
}
```

UI 展示配色：
- `problems`（发现的问题）：琥珀色（amber）
- `lessons`（踩过的坑）：红色（red）
- `improvements`（改进建议）：绿色（green）

运行中和已完成的任务均可包含反思内容。

### 7.1 三大团队

|---|---|---|---|
| 管理团队 | management | 紫色（#8b5cf6） | ~7 |
| 设计师团队 | design | 蓝色（#3b82f6） | ~4 |
| 生产团队 | production | 绿色（#22c55e） | ~7 |

全平台共 ~24 名员工，其中 ~18 名处于在岗（active）状态。

### 7.2 子团队

生产团队通过 `employees.subTeam` 列进一步细分：

- **生产管理层**：负责生产任务的调度与审核
- **内容生产层**：负责具体内容的创作与生产

### 7.3 员工状态

| 状态 | 英文标识 | 说明 |
|---|---|---|
| 在岗 | active | 正常运行中，参与任务分配 |
| 开发中 | developing | 正在培养/开发阶段 |
| 规划中 | planned | 尚在规划阶段，未上线 |
| 已停用 | inactive | 已下线，不再分配任务 |

### 7.4 IP 人设系统（EmployeePersona）

每位 AI 员工均配有完整的 IP 人设，存储于 `employees.persona` 字段（JSON 字符串），对应 `src/lib/types.ts` 中的 `EmployeePersona` 接口：

| 字段 | 类型 | 说明 |
|---|---|---|
| age | number | 年龄 |
| gender | string | 性别 |
| mbti | string | MBTI 性格类型 |
| personality | string[] | 性格标签数组 |
| catchphrase | string | 口头禅 |
| backstory | string | 背景故事 |
| workStyle | string | 工作方式 |
| interests | string[] | 兴趣爱好 |
| fashionStyle | string | 穿搭风格 |
| visualTraits | string | 视觉特征描述 |
| sceneDescription | string | 场景描述（用于 AI 肖像生成 prompt） |

解析方式：`JSON.parse(employee.persona) as EmployeePersona`

AI 肖像为 AI 生成的横版（1376×768）PNG 图片，存储于 `public/avatars/{name}.png`。当 `avatar` 为 null 时，`AiAvatar` 组件会根据员工 ID 程序化生成 SVG 机器人头像（按团队上色）。

## 9. 数据关系图

```
employees（AI 员工）
├── skills（技能）──── skill_metrics（技能指标/月）
├── metrics（绩效指标/期）
├── tasks（任务）
│   ├── task_steps（SOP 步骤）
│   └── task_outputs（任务产出）
├── version_logs（版本日志）
└── metric_configs（指标配置，可选，员工级覆盖）

metric_configs（指标配置）
├── 全局基准（employeeId = null，team = null）
├── 团队覆盖（employeeId = null，team ≠ null）
└── 员工覆盖（employeeId ≠ null）

优先级：员工覆盖 > 团队覆盖 > 全局基准
```

**关联说明：**
- 所有外键均设置 `onDelete: "cascade"`，删除员工时级联删除其所有关联数据。
- 项目未使用 Drizzle `relations()` 声明，仅使用列级 `.references()` 约束。
- `metric_configs` 与 `employees` 的关联为可选（`employeeId` 可为 null），全局和团队级配置不绑定具体员工。
